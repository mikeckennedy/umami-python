# Unit 1 — Scaffold Great Docs & write the config

**Suggested commit:** `docs: add Great Docs config for the documentation site`
**Branch:** the current `docs-exploratoin` is fine, or a fresh `docs/great-docs-site`.
**Files touched (committed):** `umami/great-docs.yml` (new), `umami/.gitignore`, `umami/pyproject.toml`.
**Files produced but NOT committed:** `umami/great-docs/` (ephemeral build dir — gitignored here).

> Read the parent [`README.md`](README.md) first — the three "Why" facts (run from `umami/`,
> `site_url` for subpath, install into the venv for dynamic introspection) explain every step.

---

## Scope

Get a working Great Docs configuration committed. No deploy, no narrative pages. After this unit,
`great-docs build` produces a correct site locally; publishing it is Unit 2.

---

## Steps

### 1. Install Great Docs into the project venv

`great-docs` is already in the `[dev]` extra (`umami/pyproject.toml`), and dynamic introspection
imports `umami`, so install the dev extra **editable into the same venv** — that gets `umami` +
`httpx2` + `pydantic` + `great-docs` together. The repo's venv is `umami-python` (auto-activates in
this shell).

```bash
# from repo root, with the umami-python venv active
uv pip install -e "./umami[dev]"   # umami (editable) + great-docs + pytest/ruff

# sanity: umami must import in this env (dynamic introspection depends on it)
python -c "import umami, httpx2, pydantic; print('umami', umami.__version__)"
great-docs --help | head -5
quarto --version        # expect 1.9.x — already installed
```

- `great-docs==0.13.0` resolves cleanly on this venv's Python 3.14.5 (verified). If a future Python
  ever breaks it, create a dedicated docs venv on 3.12/3.13 and `uv pip install -e "./umami[dev]"`
  there; record which interpreter you used in Unit 2's runbook.
- The editable install matters — see the *stale venv import trap* note in repo memory; a
  non-editable `umami` would make the docs (and tests) reflect a stale build.

### 2. Initialize from the package directory

`great-docs` must see `umami/pyproject.toml`, so run it from `umami/` (or pass `--project-path`).

```bash
cd umami
great-docs scan --verbose      # preview discovered API BEFORE writing config; note exact member names
great-docs init                # creates umami/great-docs.yml; answer "n" to its .gitignore prompt
```

- **Answer `n`** to the "Add 'great-docs/' to .gitignore?" prompt — we edit `.gitignore` ourselves in
  step 4 (the build dir is `umami/great-docs/`, so the entry belongs in `umami/.gitignore`).
- Confirm the init output reports **`Detected google docstring style`** and finds the package via
  `__all__` (≈ 26 exports). If it says numpy, that's fine — step 3 pins `parser: google` explicitly.
- Keep the `great-docs scan --verbose` output handy: it shows the **exact** names Great Docs uses for
  submodule members (e.g. whether it's `models.Website` or another form). Use those verbatim in the
  `reference:` block.

### 3. Reshape `umami/great-docs.yml` to the target config

Replace the auto-generated file with the **Target `great-docs.yml`** from
[`README.md`](README.md#target-great-docsyml). Then reconcile the `reference:` `contents:` entries
with the real names from `great-docs scan --verbose` (step 2):

- All module-level functions are top-level names (`new_event`, `login`, …) — these are safe.
- The `models.*` and `errors.*` entries are the only uncertain ones. If `scan` shows a different
  qualified form, match it. If submodule members can't be addressed individually, fall back to
  letting Great Docs auto-list them (keep the init-generated section for models/errors) rather than
  fighting the syntax.

The four lines that matter most, in priority order:

1. `site_url: "https://mkennedy.codes/docs/umami-python/"` — **must be exact, with trailing slash.**
   This is what makes subpath hosting work.
2. `parser: google`
3. `display_name: Umami Analytics`
4. `dynamic: true`

### 4. Gitignore the ephemeral build directory

Add to **`umami/.gitignore`** (not the repo-root one — the build dir is `umami/great-docs/`):

```gitignore
# Great Docs ephemeral build output (regenerated by `great-docs build`)
/great-docs/
```

### 5. Add a Documentation URL to `pyproject.toml`

In `umami/pyproject.toml`, under `[project.urls]`, add a `Documentation` entry. This surfaces on PyPI
and is a secondary `site_url` source Great Docs reads.

```toml
[project.urls]
Homepage = "https://github.com/mikeckennedy/umami-python"
Tracker = "https://github.com/mikeckennedy/umami-python/issues"
Source = "https://github.com/mikeckennedy/umami-python"
Documentation = "https://mkennedy.codes/docs/umami-python/"
```

### 6. Smoke-build (local only — do not commit output)

```bash
cd umami
great-docs build
```

Expect "Build complete — N/N steps" and `umami/great-docs/_site/index.html`. If the API-reference
step fails under dynamic introspection, Great Docs auto-retries with static analysis; if it still
fails, set `dynamic: false` in `great-docs.yml` and note it. (Full preview/asset verification is
Unit 2.)
